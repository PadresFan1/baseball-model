MLB MODEL ARCHITECTURE PROGRESS SUMMARY & NEXT CHAT CONTEXT

1. THE CORE BREAKTHROUGH (WHY WE UPGRADED)
--------------------------------------------------------------------------------
- The Problem: Our original top-down model used team-level rolling averages 
  (d_woba, d_xwoba_bat, d_xfip, d_xwoba_pit). While tightening filters increased 
  ROI, the overall win rate remained stuck around 40-44%. The closing market 
  lines had completely digested team-level trends, turning our model into an 
  accidental "underdog sniper" rather than a highly accurate predictor.
- The Solution: We tore down the top-down framework and built a highly granular 
  player-level, bottom-up model. Instead of treating teams as monolithic blocks, 
  the model now simulates every single game plate appearance by matchup.

2. UPDATED DATA ARCHITECTURE & FEATURE ENGINEERING
--------------------------------------------------------------------------------
We preserved our core four sabermetrics but completely re-mapped how they are compiled:
- Batter Metrics (d_woba, d_xwoba_bat): Tracked by Platoon Splits (vs. LHP and vs. RHP). 
  We calculate each starting batter's individual rolling 100 Plate Appearance (PA) 
  profile strictly against the handedness of tonight's starting pitcher.
- Pitcher Metrics (d_xfip, d_xwoba_pit): Tracked by tonight's specific starting 
  pitcher's individual rolling 50–70 innings pitched. 
- Lineup Weighting: We abandoned raw unweighted averages (which let bench players 
  dilute superstars). Claude Code implemented a Static Lineup PA-Weighted Aggregation 
  Matrix, weighting individual player metrics by the strict, historical probability 
  of their spot in the batting order (e.g., Slot 1 = 12.3% of game PAs, Slot 9 = 10.0%).
- The Log5 Matchup Grid: Individual batter (B) and pitcher (P) metrics are run 
  through an Odds Ratio/Log5 calculation against the baseline league average (L) 
  to project individual matchup probabilities:
  Pr = (B * P / L) / [ (B * P / L) + ((1 - B) * (1 - P) / (1 - L)) ]

3. ADVANCED EDGE CASE MODELING
--------------------------------------------------------------------------------
- Opener & Bullpen Games: To close a massive structural leak, we moved from tracking 
  a "Monolithic Starter" to an Inning Package framework. If a starter averages less 
  than 2.0 Innings Per Appearance (IPA), the engine flags an opener and weights the 
  pitching feature by realistic game exposure chunks:
  Projected xFIP = (1/9 * Opener xFIP) 
                   + (4.5/9 * Bulk Reliever xFIP) 
                   + (3.5/9 * Core Bullpen xFIP)
- Flawless IP Tracking: We eliminated estimation math. Claude Code successfully 
  updated pull_historical_lineups to pull the exact inningsPitched string from the 
  MLB API box score stats block, parsing baseball notation strings (like '5.2') 
  into exact decimals (5.667) via ip_to_float().

4. BACKTEST RESULTS & THE DUAL-MODEL STRATEGY
--------------------------------------------------------------------------------
When we ran the new player-level meta model over the 2022–2025 walk-forward seasons, 
the model immediately began heavily trusting our metrics over the market line. 
- The Sniper Corridor: At a tight >5% Edge Filter (C=1.0), the model weaponized 
  into a premium sniper rifle—yielding a massive 57.0% overall win rate and an 
  +8.3% 4-year ROI. However, the strict filters limited volume to roughly 30 
  bets per season.

The Two-Pronged Production Portfolio:
To capture more daily market inefficiencies without losing our premium edge, we 
decided to run two models side-by-side inside the same script pipeline:

  1. Model A: "The Sniper"
     - Configuration: Tight regularization (C = 1.0), Edge Filter Cutoff > 5.0%
     - Volume/Target: ~30 bets per season, high-conviction, larger unit sizing.
  2. Model B: "The Enforcer"
     - Configuration: Relaxed regularization (C = 5.0), Edge Filter Cutoff > 3.5%
     - Volume/Target: ~120–150 bets per season, capturing smaller daily market 
       mistakes at standard unit sizing.

5. AUTOMATION, ARCHIVING, & SHEET SYNCHRONIZATION
--------------------------------------------------------------------------------
Claude Code built an automated daily execution routine with strict production guidelines:
- Dual-Logging: The script checks daily lines/lineups once, evaluates both models, 
  prints a clear terminal dashboard tagging bets as [ACTION: SNIPER BET] or 
  [ACTION: ENFORCER BET], and appends them as separate rows to a local tracker.
- Local Archiving: On startup, the script automatically archives old predictions_log.csv 
  and .xlsx files with a date timestamp suffix into a backup subfolder, starting 
  a clean file for the current cycle.
- Google Sheets Value-Only Reset: The upload pipeline clears out the existing 
  projection data using a value-only wipe (batch_clear or clear targeting the cell 
  values below the header, e.g., A2:Z2000). This completely wipes the data rows for 
  the fresh run while keeping all background colors, custom borders, and 
  conditional formatting rules 100% intact.
